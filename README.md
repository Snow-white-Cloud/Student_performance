# Student_performance
Тестовое задание на позицию стажёра


## Инструкция по запуску

1. Клонируйте репозиторий:\
  ``` git clone git@github.com:Snow-white-Cloud/Student_performance.git ```\
  или скачайте его ("Download ZIP") и распакуйте.

2. В корневой папке проекта воспользуйтесь docker-compose.yml:\
   для запуска сервера с поднятием Базы Данных: ```docker compose up web```, затем перейдите на http://0.0.0.0:8000 \
   для запуска тестов (без разворачивания Базы Данных): ```docker compose run --no-deps tests```

Убедитесь, что Docker установлен и запущен, а также имена контейнеров не перекликаются с Вашими!

Автоматическая документация расположена по адресу http://0.0.0.0:8000/swagger/

## Использованные решения (особенности)

### Хранение данных

Вместо одной таблицы предоставлено разделение на `Students` и `Grades`. Данное решение обусловлено:
1) нормализацией до усиленной 3НФ или же "Нормальной формой Бойса-Кода", которая является эталоном проектирования БД;
2) стремлением уменьшить объём расходуемой БД памяти.

Было решено добавить автоинкремент `id` в качестве первичного ключа для студентов вместо уникального сочетания ФИО+группа, чтобы ускорить поиск. В противном случае для `Grades` пришлось бы хранить составной внешний ключ, что привело бы к усложнению таблицы, увеличению её объёма, замедлению поиска и дублированию. Однако, исходя из предоставленных данных, сочетание ФИО+группа действительно является уникальным, иначе они не имели бы смысла, поэтому на соответствующие поля наложено дополнительное ограничение на уникальность. Такой подход вызвал у меня недоумение, так как в жизни вполне возможны случаи полных совпадений ФИО и групп у двух разных людей.

Тип ФИО - `VARCHAR(100)`: 100 взято приблизительно, хотя при разработке настоящей системы, а не тестового задания, стоило бы изучить статистику по среднему количеству букв в полных инициалах. ФИО хранится цельно, так как по заданию может восприниматься неделимым элементом, тем самым не нарушая 1НФ. К тому же цельное хранение облегчает логику поиска.

Группа задана типом `CHAR(4)`, так как в исходных данных все номера заданы строго тремя цифрами и одной буквой.

Было решено добавить автоинкремент `id` в качестве первичного ключа для отметок, так как ни один иной атрибут или их сочетание не определяли запись однозначно.

В тесте данные определены как SCD1 (возможность редактирования), хотя за неимением потребности (и отсутствием реализации) изменять и удалять записи можно рассматривать их как SCD0. В любом случае, SCD1 и SCD0 не предполагают сохранения истории, так что избрано каскадное удаление.

При кратном увеличении объёма данных можно использовать на полях `id` в обеих таблицах индексы B-Tree для ускорения и оптимизации. При батчевой загрузке же стоит позаботиться об их удалении и последующем новом добавлении.

### Использование SQL

В данной работе логика получения аналитики вынесена в отдельные SQL-функции для демонстрации знаний, так как именно на SQL делала упор. `grades_more_than` считается мною "эталонной", т.е. наиболее оптимизированной, когда как `grades_less_than` служит пособием, демонстрирующим "как можно, но не нужно". Аналитика представлена функциями с возможностью кастомизировать запросы: узнавать не только об отметках "2", но и любых других. Требуемые заданием значения передаются по умолчанию. Однако ж достаточным вариантом являлось бы написание несложных запросов, скрытых в теле функций, без возможности пользователем ввести свои предпочтения:
```
...
query = """
SELECT s.full_name, COUNT(*) AS count_grades
        FROM Grades g
        INNER JOIN Students s ON g.id_student = s.id
        WHERE g.grade = 2
        GROUP BY s.full_name
        HAVING COUNT(*) > 3
"""
data = await connect.fetch(query)
...
```

Избрана характеристика STABLE, так как функции не изменяют БД и возвращают одно и то же значение в рамках одной сессии, что обусловлено чтением из таблиц.

Избран INNER JOIN, т.к. для вывода важны только существующие ученики и существующие отметки.

### Сервис

#### Настройки и основа

1. `config.py` содержит основные настройки сервиса: имя, описание, настройки покдючения к БД, настройки батчей, уровень логирования, загрузка констант окружения.
2. `logger_config.py`содержит настройки логирования: формат вывода сообщений, формат времени, задаёт уровень логирования из общих настроек. Вынесен в отдельный файл
   по смысловому разделению и для упрощения работы с сервисом.
3. `database.py` содержит настройку БД (запуск SQL-файлов) и функции манипуляций с пулом подключений. Выбрана реализация именно пула во имя оптимизации и избежания затрат ресурсов на
   создания подключений.
4. `main.py` - сердце сервиса. Описано поведение при запуске и закрытии, домашняя страница, подключены роутеры. 

#### Роутеры

1. `get.py` содержит логику для страниц /students/more-than-3-twos и /students/less-than-5-twos. Обращается к БД и выводить имена студентов и количество их отметок
   по заданным параметрам. Если данных нет, то возвращается пустой список `[]`. Есть обработка возможных ошибок.
2. `upload.py` содержит логику для страницы /upload-grades. Здесь происходит загрузка данных в БД из csv-файла. Выбрана батчевая загрузка, т.к. такой подход позволяет удобно валидировать
   данные и оптимизировать функцию за счёт уменьшения количества обращений к БД (снижения нагрузки на неё). Для этих же целей введён словарь `students_cache`,
   снимающий множественные запросы к БД о наличии или отсутствии студента. Это позволительно ввиду немногочисленности обучающихся (по статистике в одном ВУЗе обучаются одновременно
   не более 50.000 человек). Однако по этой же причине можно переписать логику загрузки студентов на построчную вставку (без использования батча):
   ```
   id = await connect.execute("""
     INSERT INTO Students (full_name, study_group) VALUES ($1, $2)
     RETURNING id
     """,
     full_name, study_group)
   ```
   хотя сама к такому подходу отношусь скептично. Загрузка для отметок известных студентов и загрузка для неизвестных БД студентов и их отметок разнесены в отдельные функции,
   дабы избежать дублирования кода. Есть обработка возможных ошибок.

#### Безопасность и окружение

Безопасность обеспечивается Pedantic-схемами, дополнительными проверками во время работы приложения и логированием. Табуировано использование f-строк при работе с БД, что исключает опасность SQL-инъекций. Вставки записей в БД происходят внутри транзакций. Ошибки отлавливаются и обрабатываются. Важные настройки (логин, пароль и т.д.) хранятся в переменных окружения, а не в самом коде. 
1. `schemas.py` содержит различные Pedantic-схемы для валидации поступающих и исходящих данных.
2. `depends.py` содержит зависимости для получения валидных параметров для страниц /students/more-than-3-twos и /students/less-than-5-twos.
3. `db.env` - файл окружения.

#### Контейнеризация

В `docker-compose.yml` описываются три сервиса: `db` (`container_name: database`) поднимает БД, `web` (`container_name: web`) запускает сервис, а `tests` (`container_name: tests`) отвечает за тесты к сервису. Исполнитель - `Dockerfile` в корне проекта. Зависимости (библиотеки) описаны в файле `requirements.txt`.

### Тестирование

Полностью расположено в папке `tests`. Для их реализации разработаны фикстуры для моков `fixture.py` и класс-имитация БД `imitationDB.py`. В папке `data` расположены варианты загружаемых файлов.

Слой ЮНИТ.\
Тесты этого слоя охватывают только модуль `upload`. Они проверяют корректность поведения функций `flush_batch_grades_know_stud` и `flush_batch_grades_unknow_stud` при различных ответах от БД. Используются моки. Файлы: `test_unit_upload.py`

Слой ИНТЕГРАЦИЯ.\
Тесты этого слоя охватывают два модуля: `upload` и `get`. Они проверяют ответы сервиса. Для первого модуля используются класс-имитация БД, т.к. важно проверить изменчивость данных при вставке; для второго - достаточно моков подключения к БД и "фальшивых" откликов. Файлы: `test_integration_upload.py` и `test_integration_get.py`.


## Неоптимизированные или проблематичные места

Однако даже при внимательном отношении к проекту остались пробелы. Некоторые возможные изменения уже были предложены мною ранее. Остальные таковы.

1. Проблемная загрузка. Если соединение с БД отвалится при выполнении функции `flush_batch_grades_unknow_stud` между выполнениями транзакций, то нынешняя реализация не сможет это учесть.
   Возможных решения два: либо переписать логику загрузки в одну единую транзакцию, либо выводить пользователю обратную связь о том, какие именно записи были успешно загружены.
2. Усложняется логика из-за использования SQL-функций за счёт манипуляций с JSON.
3. Избыточность: в решении представлена избыточность, которая при развитии проекта могла бы дать хорошую площадку для расширения возможностей аналитики, однако в данной конкретной
   задаче может быть засчитана за лишнее усложнение.
